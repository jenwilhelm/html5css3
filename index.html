goals of html5

  • Support existing content
  • Specify how browsers should handle invalid markup
  • Evolve the language to support the development of web apps (APIs)

what's different?

html5's syntax is a lot more lax than xhtml 1.0. Whereas xhtml 1.0 enforced lowercase tags and attributes, quoted attributes, and closing tags (including standalone elements, such as <br>), html5 does not.

						
<-- valid xhtml 1.0 -->
<input type="radio" checked="checked" />


<-- invalid xhtml 1.0, valid html5 -->
<INPUT TYPE=RADIO CHECKED>


<-- valid html5 -->
<input type="radio" checked>

byebye, type

The "type" attribute is no longer needed in css or script declarations.

						
<link rel="stylesheet" href="css/screen.css" 
type="stylesheet">

<script type="text/javascript" src="js/yay.js" 
type="text/javascript"></script>

can now become:

						
<link rel="stylesheet" href="css/screen.css">

<script src="js/yay.js"></script>

obsolete (not deprecated)

While you can still use these elements and attributes, html5 considers the following obsolete:

  • frame
  • font
  • bgcolor
  • cellpadding, cellspacing
  • acronym (use abbr instead)
  • center

Use css instead!

the doctype

Traditionally, a doctype is used to tell the browser which version of markup the page was written in, such as html 4.01 or xhtml 1.0.

You may be used to seeing something like this:

						
<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01 
Transitional//EN">

Because html5 aims to be backwards-compatible and support all future versions of html, there is no need for a version number.

With html5, you can now simply use:

						
<!doctype html>

Much easier to remember, no?

Sites that use the html5 doctype include Zappos, Google, YouTube, & Twitter.

structural elements

html5 introduces some commonly-used structural elements for more meaningful & semantic markup.

  • nav
  • section
  • header
  • footer
  • aside
  • article

nav

An element to be used as a navigational aid – usually a set of links.

						
<nav id="mainNav">
  <a href="home.html">Home</a>
  <a href="about.html">About</a>
  <a href="contact.html">Contact</a>
  <a href="cupcakes.html">Cupcakes</a>
</nav>

section

Used for grouping together related content.

						
<section>
  <h1>Cupcakes</a>
  <p>What makes cupcakes better than muffins?
  Everything.</p>
  <p><a href="bit.ly/9rPHp0">Best cupcakes
  ever?</a></p>
</section>

header

An element used to contain "a group of introductory or navigational aids". This does not necessarily mean it can be used for mastheads only – it can also be used in the <section> element, for example.

						
<header>
  <div id="logo"></div>
  <nav>
    <a href="home.html">Home</a>
    <a href="about.html">About</a>
    <a href="contact.html">Contact</a>
    <a href="cupcakes.html">Cupcakes</a>
  </nav>
</header>

...

<section>
  <header>
    <h1>Cupcakes</h1>
  </header>
  <p><a href="bit.ly/9rPHp0">Best cupcakes 
  ever?</a></p>
</section>

footer

Used for grouping together information about its parent element. Think metadata. As with the <header> element, <footer> can be used for many different parent elements.

						
<footer>
  <nav>
    <a href="copyright.html">Copyright</a>
    <a href="legal.html">Legal</a>
    <a href="contact.html">Contact</a>
  </nav>
</footer>

...

<section>
  <header>
    <h1>Cupcakes</h1>
  </header>
  <p><a href="bit.ly/9rPHp0">Best cupcakes
  ever?</a></p>
  <footer>
    <p>By Jen Wilhelm</p>
  </footer>
</section>

aside

An element used for content that is separate from the main content (pullquotes, advertising, sidebars, etc).

						
 <aside>
  <h2>Cupcake of the day</h2>
  <p><a href="bit.ly/fjCrEk">Chocolate 
  cheesecake</a></p>
 </aside>

article

An element used for "self-contained" content. This can be easily confused with the <section> element. Think: can this content be syndicated?

						
<article>
  <header>
    <h1>Top 3 best cupcakes ever</h1>
  </header>
  <p>These are my top 3 favorite cupcakes ever.
  What are yours?</p>
  <ul>
    <li>Chocolate cheesecake</li>
    <li>Red velvet</li>
    <li>Vanilla</li>
  </ul>
  <footer>
    <p>3 comments</p>
    <p>Author: Jen Wilhelm</p>
    <p>Published: January 14, 2011</p>
  </footer>
</article>

content models

Previous versions of html divided elements into two categories: inline and block-level.

html5 divides them up even further. Elements that were previously inline are now considered "text-level". Many block-level elements fall under "grouping content". Forms have their own, and images/audio/video/canvas are "embedded." All of html5's new elements (header, section, article, etc) are under a new content model called "sectioning content."

sectioning content

Say we have the following:

						
<h1>Jen's favorite things*</h1>
  <h2>Cupcakes</h2>
    <h3>Chocolate cheesecake</h3>
    <h3>Red Velvet</h3>
    <h3>Vanilla</h3>
  <h2>White things</h2>
    <h3>Snow</h3>
    <h3>Marshmallows</h3>
  <h2>Cleaning</h2>
<small>* Not an inclusive list</small>

This will yield:

  • Jen's favorite things*
    • Cupcakes
      • Chocolate cheesecake
      • Red Velvet
      • Vanilla
    • White things
      • Snow
      • Marshmallows
    • Cleaning

But what about that <small> element? Which element is it associated with? Browsers have no way of knowing that it should be associated with the entire document, not just the last <h2> element. Sectioning elements to the rescue! Let's try this instead:

						
<h1>Jen's favorite things*</h1>
  <section>
  <h2>Cupcakes</h2>
    <h3>Chocolate cheesecake</h3>
    <h3>Red Velvet</h3>
    <h3>Vanilla</h3>
  <h2>White things</h2>
    <h3>Snow</h3>
    <h3>Marshmallows</h3>
  <h2>Cleaning</h2>
  </section>
<small>* Not an inclusive list</small>

Now the browser understands that the <small> element falls under the <h1>, not the <h2>. If you wanted to go even further, then each <h2> element could be placed in its own <section>.

outline algorithm

html5 uses a new outline algorithm, which states that each piece of sectioning content has its own self-contained outline.

						
<h1>Jen's favorite things*</h1>
  <section>
  <h1>Cupcakes</h1>
    <h2>Chocolate cheesecake</h2>
    <h2>Red Velvet</h2>
    <h2>Vanilla</h2>
  <h1>White things</h1>
    <h2>Snow</h2>
    <h2>Marshmallows</h2>
  <h1>Cleaning</h1>
  </section>
<small>* Not an inclusive list</small>

This will produce the same outline:

  • Jen's favorite things*
    • Cupcakes
      • Chocolate cheesecake
      • Red Velvet
      • Vanilla
    • White things
      • Snow
      • Marshmallows
    • Cleaning

hgroup

Sometimes you might want to use a header element, but you don't want it to show up in the document outline.

						
<h1>html5 & css3 for noobs</h1>
<h2>and other fun things, too!</h2>
  • html5 & css3 for noobs
    • and other fun things, too!

By enclosing headers with <hgroup>, we can tell the document to ignore the enclosed headings in the outline.

						
<h1>html5 & css3 for noobs</h1>
<hgroup>
  <h2>and other fun things, too!</h2>
</hgroup>

new input types

  • email
  • url
  • tel
  • range
  • number
  • color
  • dates & times (date, datetime, datetime-local, time, month)

email

						
<input type="email" name="userEmail">

See what happens in Google Chrome when you type an invalid email first in a type="text" field, then a type="email" field:

url

						
<input type="url" name="twitterUrl">

No noticeable difference in Chrome, but if a user types an invalid URL, Opera will give a list of recently visited URLs.

tel

						
<input type="tel" name="homePhone">

range

						
<input type="range" name="homePhone" min="1" max="10">

Please rate this cupcake:

number

						
<input type="number" name="howManyCupcakes" min="-1" max="20">

How many cupcakes have you eaten today?

date

						
<input type="date" name="birthday">

Browsers that don't understand the new input types will default to type="text".

Browsers will also treat styles and handle errors with the new input types differently.

Even if there is no visual difference in the input type (text vs. email), using the semantically correct type now may create an enhanced experience once support is more widely available. Example: mobile Safari will produce the email keyboard when presented with an email input type.

new attributes

  • placeholder
  • autofocus
  • required
  • pattern
  • autocomplete

placeholder

Placeholder text allows you to give an input a temporarily value – commonly used for describing what should be entered – and will show/hide depending on the value that's entered.

						
<input type="email" name="userEmail" 
placeholder="name@company.com">

autofocus

Autofocus forces an input field to automatically be focused on after page load. This can be annoying for power users who commonly use their keyboard to navigate the web. Do not abuse this property.

						
<input type="url" name="searchBar" autofocus>

required

Required forces a user to enter in required fields before the form can be submitted.

						
<input type="text" name="favoriteCupcake" 
placeholder="Chocolate cheesecake" required>

pattern

What happens if html5 doesn't offer a particular input type (zip code), but you need to validate the value on the client-side? Don't cry – pattern is here to save the day. But, you need to understand regular expressions. So go ahead and cry.

						
<input type="text" name="zipCode" pattern="[\d]{5}(-[-d]{4})" required>

autocomplete

Autocomplete allows you to turn on/off the browser's default way of handling form auto-completion. This can be useful for especially sensitive information, such as credit card numbers and social security numbers. It can hold only two possible values: "on" or "off".

						
<input type="text" name="ccNumber" required autocomplete="off">

canvas

What exactly is canvas? Simply put, it's an element for creating dynamic images.

						
<canvas id="canvas" width="300" height="300">
  Your browser does not support canvas. Sad. :(
</canvas>

The rest of the work is done with JavaScript.

						
var canvas = document.getElementById('canvas'),
    context = canvas.getContext('2d');

// stroke black
context.strokeStyle = '#000';

// left, top, width, height
strokeRect(0, 0, 200, 100);

The canvas 2d API also provides lots of other methods: fillStyle, fillRect, lineWidth, etc.

audio & video

Previous to html5, web developers had to rely on third-party technology to share audio & video (Flash, RealPlayer, etc).

audio

						
<audio src="shotsshotsshots.mp3" autoplay loop 
controls></audio>

But what if your browser doesn't support .mp3 format?

						
<audio autoplay loop controls>
  <source src="shotsshotsshots.ogg">
  <source src="shotsshotsshots.mp3">
</audio>

video

						
<video src="lotr.mp4" width="360" height="240" 
controls poster="lotr.jpg"><h;/video>

Again, what happens if your browser doesn't support .mp4?

						
<video autoplay loop controls>
  <source src="lotr.ogv" type="video/ogg">
  <source src="lotr.mp4" type="video/mp4">
</video>

browser prefixes

Browser prefixes allow browsers to start implementing parts of the spec before it's finished. It allow allows vendors to play with experimental properties that are not part of the spec.

Browser Prefix
Safari -webkit-
Mozilla -moz-
Opera -o-
Microsoft -ms-

When specifying a css3 property, order matters. Place the non-prefixed version last so that once a browser does implement a property according to the spec, the real property will trump the prefixed property.

Before css3, rounded corners were, to put it lightly, a pain in the ass. There were tons of different methods invented in order to achieve the rounded corner effect, the most popular one being the use of images.

If you knew the width of your element would not change, you could do something like this:

						
<div class="roundedCornerTop"></div>
<div class="roundedCornerMiddle">
  Content here
</div>
<div class="roundedCornerTop"></div>

...

.roundedCornerTop {
  width: 200px;
  background: url("images/cornerTop.png") 
    top left no-repeat;
}

.roundedCornerMiddle {
  width: 200px;
  background-color: #ed7624;
}

.roundedCornerBottom {
  width: 200px;
  background: url("images/cornerBottom.png") 
    top left no-repeat;
}

roundedCornerMiddle

selectors

css3 introduces lots of helpful new selectors.

E[att^=”val”]
Matches any E element whose att attribute value begins with “val”.

E[att$=”val”]
Matches any E element whose att attribute value ends with “val”.

E[att*=”val”]
Matches any E element whose att attribute value contains the substring “val”.

E:nth-child(n)
Matches any E element that is the n-th child of its parent.

E:nth-of-type(n)
Matches any E element that is the n-th sibling of its type.

E:first-child
Matches any E element that is the first child of its parent.

E:last-child
Matches any E element that is the last child of its parent.

E:only-child
Matches any E element that is the only child of its parent.

E:checked, E:disabled, E::selection
Matches any user interface element (form control) E that is checked/disabled/selected.

E:not(s)
Matches any E element that does not match the simple selector s.

examples

						
/* Selects all form  inputs of type="text" */
form input[type=text] { ... }

/* Selects all even table rows */
table tr:nth-child(even) { ... }

/* Selects all even table rows */
table tr:nth-child(odd) { ... }

/* Selects every third table row */
table tr:nth-child(3n+3) { ... }

/* Selects the first and last list elements */
ul li:first-child { ... }
ul li:last-child { ... }

There's even a handy little chart that shows which selectors are supported in what browser.
