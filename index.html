goals of html5

  • Support existing content
  • Specify how browsers should handle invalid markup
  • Evolve the language to support the development of web apps (APIs)

what's different?

html5's syntax is a lot more lax than xhtml 1.0. Whereas xhtml 1.0 enforced lowercase tags and attributes, quoted attributes, and closing tags (including standalone elements, such as <br>), html5 does not.

						
							<-- valid xhtml 1.0 -->
							<input type="radio" checked="checked" />

							
							<-- invalid xhtml 1.0, valid html5 -->
							<INPUT TYPE=RADIO CHECKED>

							
							<-- valid html5 -->
							<input type="radio" checked>

byebye, type

The "type" attribute is no longer needed in css or script declarations.

						
							<link rel="stylesheet" href="css/screen.css" type="stylesheet">

							<script type="text/javascript" src="js/cupcakes.js" type="text/javascript"></script>

can now become:

						
							<link rel="stylesheet" href="css/screen.css">

							<script src="js/cupcakes.js"></script>

obsolete (not deprecated)

While you can still use these elements and attributes, html5 considers the following obsolete:

  • frame
  • font
  • bgcolor
  • cellpadding, cellspacing
  • acronym (use abbr instead)
  • center

Use css instead!

the doctype

Traditionally, a doctype is used to tell the browser which version of markup the page was written in, such as html 4.01 or xhtml 1.0.

You may be used to seeing something like this:

						<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01 Transitional//EN">

Because html5 aims to be backwards-compatible and support all future versions of html, there is no need for a version number.

With html5, you can now simply use:

						<!doctype html>

Much easier to remember, no?

Sites that use the html5 doctype include Zappos, Google, YouTube, & Twitter.

structural elements

html5 introduces some commonly-used structural elements for more meaningful & semantic markup.

  • nav
  • section
  • header
  • footer
  • aside
  • article

nav

An element to be used as a navigational aid – usually a set of links.

						
<nav id="mainNav">
  <a href="home.html">Home</a>
  <a href="about.html">About</a>
  <a href="contact.html">Contact</a>
  <a href="cupcakes.html">Cupcakes</a>
</nav>

section

Used for grouping together related content.

						
<section>
  <h1>Cupcakes</a>
  <p>What makes cupcakes better than muffins?
  Everything.</p>
  <p><a href="bit.ly/9rPHp0">Best cupcakes
  ever?</a></p>
</section>

header

An element used to contain "a group of introductory or navigational aids". This does not necessarily mean it can be used for mastheads only – it can also be used in the <section> element, for example.

						
<header>
  <div id="logo"></div>
  <nav>
    <a href="home.html">Home</a>
    <a href="about.html">About</a>
    <a href="contact.html">Contact</a>
    <a href="cupcakes.html">Cupcakes</a>
  </nav>
</header>

...

<section>
  <header>
    <h1>Cupcakes</h1>
  </header>
  <p><a href="bit.ly/9rPHp0">Best cupcakes 
  ever?</a></p>
</section>

footer

Used for grouping together information about its parent element. As with the <header> element, <footer> can be used for many different parent elements. Think metadata.

						
<footer>
  <nav>
    <a href="copyright.html">Copyright</a>
    <a href="legal.html">Legal</a>
    <a href="contact.html">Contact</a>
  </nav>
</footer>

...

<section>
  <header>
    <h1>Cupcakes</h1>
  </header>
  <p><a href="bit.ly/9rPHp0">Best cupcakes
  ever?</a></p>
  <footer>
    <p>By Jen Wilhelm</p>
  </footer>
</section>

aside

An element used for content that is separate from the main content (pullquotes, advertising, sidebars, etc).

						
 <aside>
  <h2>Cupcake of the day</h2>
  <p><a href="bit.ly/fjCrEk">Chocolate 
  cheesecake</a></p>
 </aside>

article

An element used for "self-contained" content. This can be easily confused with the <section> element. Think: can this content be syndicated?

						
<article>
  <header>
    <h1>Top 3 best cupcakes ever</h1>
  </header>
  <p>These are my top 3 favorite cupcakes ever.
  What are yours?</p>
  <ul>
    <li>Chocolate cheesecake</li>
    <li>Red velvet</li>
    <li>Vanilla</li>
  </ul>
  <footer>
    <p>3 comments</p>
    <p>Author: Jen Wilhelm</p>
    <p>Published: January 14, 2011</p>
  </footer>
</article>

form elements

Jen's favorite part of html5. This stuff is pretty awesome.

new input types

  • email
  • url
  • tel
  • range
  • number
  • color
  • dates & times (date, datetime, datetime-local, time, month)

email

						
<input type="email" name="userEmail">

See what happens in Google Chrome when you type an invalid email first in a type="text" field, then a type="email" field:

url

						
<input type="url" name="twitterUrl">

No noticeable difference in Chrome, but if a user types an invalid URL, Opera will give a list of recently visited URLs.

tel

						
<input type="tel" name="homePhone">

range

						
<input type="range" name="homePhone" min="1" max="10">

Please rate this cupcake:

number

						
<input type="number" name="howManyCupcakes" min="-1" max="20">

How many cupcakes have you eaten today?

date

						
<input type="date" name="birthday">

Browsers that don't understand the new input types will default to type="text".

Browsers will also treat styles and handle errors with the new input types differently.

Even if there is no visual difference in the input type (text vs. email), using the semantically correct type now may create an enhanced experience once support is more widely available. Example: mobile Safari will produce the email keyboard when presented with an email input type.

new attributes

  • placeholder
  • autofocus
  • required
  • pattern
  • autocomplete

placeholder

Placeholder text allows you to give an input a temporarily value – commonly used for describing what should be entered – and will show/hide depending on the value that's entered.

						
<input type="email" name="userEmail" 
placeholder="name@company.com">

autofocus

Autofocus forces an input field to automatically be focused on after page load. This can be annoying for power users who commonly use their keyboard to navigate the web. Do not abuse this property.

						
<input type="url" name="searchBar" autofocus>

required

Required forces a user to enter in required fields before the form can be submitted.

						
<input type="text" name="favoriteCupcake" 
placeholder="Chocolate cheesecake" required>

pattern

What happens if html5 doesn't offer a particular input type (zip code), but you need to validate the value on the client-side? Don't cry – pattern is here to save the day. But, you need to understand regular expressions. So go ahead and cry.

						
<input type="text" name="zipCode" pattern="[\d]{5}(-[-d]{4})" required>

autocomplete

Autocomplete allows you to turn on/off the browser's default way of handling form auto-completion. This can be useful for especially sensitive information, such as credit card numbers and social security numbers. It can hold only two possible values: "on" or "off".

						
<input type="text" name="ccNumber" required autocomplete="off">
